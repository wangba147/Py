今日目标: 学会用 Python 启动并控制外部进程, 特别是用来编译和运行你写的 C++ 程序。 这是把 Python 当"自动化胶水"的关键一步 —— subprocess 之于 Python, 就像 system() / popen() 之于 C, 但好用十倍。

1. 为什么 C++ 开发者要学 subprocess

在 C++ 项目里, 我们经常需要写一些"外围脚本":

  • 提交前跑一遍 clang-format + clang-tidy
  • 触发 cmake --build 并把警告统计出来
  • 跑单元测试 ./unit_test --gtest_filter=... 并解析输出
  • 从 git 拉数据, 调用 protoc 重新生成代码

这些事过去用 bash / .bat + system() 拼起来也能做, 但 Windows 和 Linux 语法差异巨大, 跨平台非常痛苦。Python 的 subprocess 模块 给你一套统一、跨平台、可控的 API, 几乎可以替换所有 shell 脚本。

2. 三种启动子进程的方式

2.1 os.system —— 最朴素

C++ 里的 system(const char*) 大家都熟, Python 的 os.system 几乎一模一样: 只负责"启动 + 等结束", 拿不到 stdout, 只能拿到一个粗粒度的退出码。

// C++ 视角
#include <cstdlib>
int rc = std::system("g++ main.cpp -o main");   // 只能拿到 rc
std::cout << "rc = " << rc << std::endl;
# Python 视角
import os
rc = os.system("g++ main.cpp -o main")
print(f"rc = {rc}")
经验: os.system 在 Windows 上返回 0/1, 在 Linux 上是 shell 退出码左移 8 位 —— 跨平台代码不要依赖具体数值, 最好只判断 == 0。

2.2 os.popen —— 能读 stdout, 但只能读

对应 C 里的 popen("cmd", "r") + fread。用起来像打开文件一样:

# 类似 C++ 的:  FILE* fp = popen("g++ --version", "r"); char buf[256]; fread(...);
import os
with os.popen("g++ --version") as fp:
    output = fp.read()           # 一次性读完整个 stdout
print(output)

2.3 subprocess.run —— 现代推荐

一次性解决"启动 + 等待 + 捕获输出 + 拿到退出码 + 异常处理"。99% 的场景用它就够了。

import subprocess
r = subprocess.run(["g++", "--version"], capture_output=True, text=True)
print("stdout:", r.stdout)
print("stderr:", r.stderr)
print("returncode:", r.returncode)
为什么推荐 subprocess: 列表形式 (["g++", "--version"]) 不走 shell, 避免命令注入; capture_output=True 同时拿到 stdout/stderr; text=True 把 bytes 解码成 str, 直接打印; check=True 让非零退出码自动抛异常。

3. subprocess.run 核心参数

参数作用常见坑
args 命令, 推荐 list 形式 字符串形式默认走 shell, 拼接用户输入危险
capture_output 捕获 stdout+stderr 到 r.stdout/r.stderr 等价于 stdout=PIPE, stderr=PIPE
text=True bytes → str (用系统默认编码) 读 C++ 输出里含中文时建议加 encoding="utf-8"
check=True returncode ≠ 0 时抛 CalledProcessError 需要配合 try/except
timeout=秒数 超时强杀 超时后命令可能仍在后台, 需要手动 proc.kill()
cwd=目录 指定子进程的工作目录 替代 cd build && cmake ..
env={...} 用自定义环境变量启动 要做交叉编译时很方便
input="..." 把数据写进 stdin 要求 stdin=PIPE(run 模式自动)

3.1 shell=True 的双刃剑

# 能用管道、通配符、环境变量 —— 但要小心注入
r = subprocess.run("echo $OS" if sys.platform != "win32"
                  else "echo %OS%",
                  shell=True, capture_output=True, text=True)

# 千万别这样: 用户输入拼到 shell 字符串里
user_input = "; rm -rf /"          # 来自不可信源
subprocess.run(f"echo {user_input}", shell=True)  # 灾难!
安全铁律: 凡是要把外部数据 (用户输入、配置文件、URL) 拼进命令, 就用 列表形式 + shell=False。Windows 列表形式中, 可执行文件后跟参数即可, 不必再加 cmd /c。

3.2 check=True 配合异常处理

try:
    subprocess.run(["g++", "main.cpp", "-o", "main"],
                   check=True, capture_output=True, text=True)
    print("编译成功")
except subprocess.CalledProcessError as e:
    print(f"编译失败 rc={e.returncode}\n{e.stderr}")
except FileNotFoundError as e:
    print(f"找不到可执行文件: {e}")

3.3 timeout 与子进程残留

超时不保证子进程会消失 —— 尤其 Windows, subprocess 的 timeout 抛异常后, 子进程可能还在跑。复杂场景需要 Popen + proc.kill()。

4. 捕获 stdout / stderr / 退出码

这是 C++ 开发者最关心的: 怎么把编译器的 warning 抓出来?

import subprocess, sys

r = subprocess.run(
    [sys.executable, "-c",
     "import sys; print('out'); print('err', file=sys.stderr)"],
    capture_output=True, text=True,
)
print("stdout:", repr(r.stdout))   # 'out\n'
print("stderr:", repr(r.stderr))   # 'err\n'
print("returncode:", r.returncode) # 0

4.1 合并到同一个流

想看 "C++ 编译器把 warning 打到 stderr" 时, 经常需要让两路合并, 保留顺序:

r = subprocess.run(["g++", "main.cpp"],
                   stdout=subprocess.PIPE,
                   stderr=subprocess.STDOUT,    # 把 stderr 合并到 stdout
                   text=True)
print(r.stdout)  # stdout 和 stderr 在一起, 按写入顺序

4.2 解析 GCC/Clang 的 warning 行

import re
warnings = re.findall(r"^(.*?):(\d+):(\d+):\s+(warning|error):\s+(.*)$",
                     r.stderr, re.MULTILINE)
for file, line, col, level, msg in warnings:
    print(f"{level.upper()}  {file}:{line}:{col}  {msg}")

这一行, 等价于你以前在 C++ 里写 std::regex_search + 字符串切分 —— Day 06 的 正则技能正好用上。

5. Popen 实时流式输出

编译大项目时, 等 30 秒后突然看见一大坨报错, 体验很差。最好"边跑边打", 像 tail -f。 这时候需要 Popen + 逐行迭代 stdout:

import subprocess

proc = subprocess.Popen(
    ["cmake", "--build", "build", "-j"],
    stdout=subprocess.PIPE,
    stderr=subprocess.STDOUT,
    text=True,
    bufsize=1,        # 行缓冲
)

for line in proc.stdout:        # 子进程关闭 stdout 后, 迭代自动结束
    print("  >>", line, end="")
    # 这里可以再过滤 / 落盘 / 推送到 Web

proc.wait()                       # 必须 wait 拿退出码
if proc.returncode != 0:
    print("构建失败")
进阶技巧: 真正想做 CI 工具时, 用 threading.Thread 单独读 stdout/stderr, 避免管道 buffer 写满导致子进程卡死。Python 官方文档有详细例子。

6. 通过 stdin 喂数据给子进程

很多命令行工具是"读 stdin 干活的": bc、python -i、你自己写的 std::cin >> ... 测试程序。Python 几行就能驱动它们:

r = subprocess.run(
    [sys.executable, "-c",
     "name = input('name: '); print(f'hello {name}')"],
    input="Nova\n",
    capture_output=True, text=True,
)
print(r.stdout)  # "name: hello Nova\n"

对应 C 里写起来很痛苦:

// C 视角: 你得自己管 pipe + fork + dup2
int fds[2]; pipe(fds);
pid_t pid = fork();
if (pid == 0) {              // 子进程
    close(fds[1]);
    dup2(fds[0], STDIN_FILENO);
    execl("./myprog", "myprog", NULL);
}
close(fds[0]);
write(fds[1], "Nova\n", 5);
close(fds[1]);
waitpid(pid, NULL, 0);

7. 实战: 编译并运行 C++ 程序

把前面所有点串起来: 自动找编译器、编译、跑测试、收集结果。

import shutil, subprocess
from pathlib import Path

# C++ 里要 SearchPathW / access(..., X_OK), Python 一行
compiler = None
for cand in (["g++"], ["clang++"]):
    if shutil.which(cand[0]) is not None:
        compiler = cand
        break

src = Path("main.cpp")
exe = src.with_suffix(".exe") if sys.platform == "win32" else src.with_suffix("")

# 编译
cr = subprocess.run(compiler + ["-std=c++17", "-O2", "-Wall",
                              str(src), "-o", str(exe)],
                     capture_output=True, text=True)
if cr.returncode != 0:
    print("编译失败:", cr.stderr)
    sys.exit(1)

# 运行
rr = subprocess.run([str(exe)], capture_output=True, text=True)
print("stdout:", rr.stdout)
print("rc:", rr.returncode)

exe.unlink()  # 清理产物

7.1 处理跨平台路径

Windows 的可执行文件要加 .exe, 路径分隔符也不同。用 pathlib.Path 可以让 脚本在两个平台都跑:

from pathlib import Path, PureWindowsPath, PurePosixPath
src = Path("src") / "main.cpp"     # 跨平台都行, 自动选分隔符
print(src)                          # Windows: src\main.cpp / Linux: src/main.cpp

8. 实战: 把构建日志写到文件

CI / 自动化测试必备: 把子进程输出实时同时打到屏幕 + 文件。

from pathlib import Path
import subprocess

log = Path("build.log").open("w", encoding="utf-8")
proc = subprocess.Popen(
    ["cmake", "--build", "build", "-j"],
    stdout=subprocess.PIPE,
    stderr=subprocess.STDOUT,
    text=True, bufsize=1,
)
for line in proc.stdout:
    print(line, end="")    # 屏幕
    log.write(line)        # 文件
proc.wait()
log.close()
print(f"日志: build.log, returncode = {proc.returncode}")
经验: text=True + bufsize=1 表示行缓冲的文本模式。 Python 3.7+ 推荐用 encoding="utf-8" 显式指定编码, 避免跨平台 (Windows 中文环境用 GBK) 踩坑。

9. shlex 与跨平台命令解析

用户传进来一串命令, 你想"安全地"拆成列表, 交给 subprocess.run:

import shlex, sys
cmd = 'g++ -std=c++17 -O2 -DFOO=1 main.cpp -o main'
args = shlex.split(cmd, posix=(sys.platform != "win32"))
print(args)
# ['g++', '-std=c++17', '-O2', '-DFOO=1', 'main.cpp', '-o', 'main']
Windows 上 posix 模式要设为 False, 否则引号解析规则不对 (Windows 用双引号包裹路径, 路径里可以含空格但不能含 "。posix 模式下反了)。

10. 安全与常见坑

10.1 跨平台命令差异

差异点WindowsLinux/macOS
可执行后缀.exe无
查找命令wherewhich
环境变量引用%PATH%$PATH
路径分隔符;:
默认编码GBK / CP936UTF-8
杀死进程taskkill /F /PID xxxkill -9 xxx

最佳实践: 用 pathlib.Path、os.environ、shutil.which, 而不是自己拼路径和命令。

10.2 死锁 (Deadlock)

常见 bug: capture_output=True + 不读输出, 子进程的管道 buffer (通常 4KB) 写满就阻塞。

# 错误: 这个程序可能永远卡住
proc = subprocess.Popen(["yes"], stdout=subprocess.PIPE)
proc.wait()                # yes 会一直写, 没人读, 死锁!

# 正确: 读掉 / 或丢到 DEVNULL
proc = subprocess.Popen(["yes"], stdout=subprocess.DEVNULL)
proc.wait()

10.3 进程组与会话

有时子进程会 fork 孙子进程 (比如 make -j 启动的 worker), 直接 proc.kill() 只杀父进程, 孙子成为孤儿。用 Popen(start_new_session=True) 启动新会话, 然后用 os.killpg() 整组杀。

12. 小结与练习

本节核心知识

  • os.system / os.popen 都能用, 但 推荐 subprocess.run
  • 列表形式 + shell=False 是安全姿势, 拼接用户输入必须用
  • capture_output=True + text=True 一行拿到 stdout/stderr
  • check=True + timeout= + CWD + env= 是四大控制开关
  • 实时输出要 Popen + 逐行迭代 stdout
  • shlex.split 可以安全地把字符串命令拆成 list
  • 跨平台记住: 显式 encoding="utf-8"、shutil.which、pathlib
  • 死锁是新手最常踩的坑: 不读输出 + 满 buffer

练习

  1. 写一个脚本, 找当前目录所有 .cpp 文件, 逐个调用 g++ -fsyntax-only 做语法检查, 统计每个文件有几条 warning。
  2. 扩展 cpp_build_runner.py: 支持 --rerun-on-change, 用文件 mtime 判断是否需要重新编译。
  3. 写一个跨平台的 "process info" 工具: 给定 PID, 打印进程名、命令行、占用内存。 Windows 用 wmic, Linux 用 ps。把结果合并成统一的 JSON。
  4. 挑战: 写一个 Popen 包装类, 同时在屏幕 + 日志文件实时输出, 并支持 Ctrl-C 杀掉整个进程组。
