2. 项目脚手架自动生成

每次新建 C++ 项目，都要创建一堆目录和模板文件。Python 用 pathlib + string.Template 一键搞定。

C++ 手动方式
// 手动 mkdir、touch、编写 CMakeLists.txt
// 每个项目重复相同的创建流程
// 需要记住每个模板格式

// 典型项目结构：
project/
├── src/main.cpp      // 手写
├── include/project.h // 手写
├── tests/            // 手建
├── CMakeLists.txt    // 手写
├── .gitignore        // 手写
└── README.md         // 手写
Python 自动生成
from pathlib import Path
from string import Template

# 用 Template 定义模板
cmake_tpl = Template("""
cmake_minimum_required(VERSION 3.16)
project($name VERSION $version LANGUAGES CXX)
set(CMAKE_CXX_STANDARD $cxx_std)
""")

# 一键创建目录结构
dirs = ["src", "include", "tests", "docs"]
for d in dirs:
    Path(project_dir / d).mkdir(parents=True)

# 模板填充 → 生成文件
(Path(project_dir / "CMakeLists.txt")
 .write_text(cmake_tpl.substitute(
     name="my_project",
     version="0.1.0",
     cxx_std="17")))

关键要点

Template vs f-string：模板中的 $name 不会意外被 Python 变量替换，比 f-string 更安全（尤其是生成 C++ 代码时 ${SOURCES} 是 CMake 变量，不是 Python 变量）。

pathlib vs os：Day 03 学过的 pathlib.Path 比 os.path 更直观——Path("/src") / "main.cpp" vs os.path.join("/src", "main.cpp")。

注意 Template.safe_substitute()：未匹配的 $var 会保留原样（不报错），适合生成 C++ 代码模板。而 Template.substitute() 遇到未定义变量会抛 KeyError。

3. 代码度量分析

量化代码质量是 C++ 项目管理的刚需。Python 用正则 + Counter 轻量统计，无需安装 cppcheck 等重型工具。

PYTHON — 代码度量核心逻辑
import re
from collections import Counter

# 函数数量统计
func_re = re.compile(
    r'(?:inline\s+)?(?:static\s+)?(?:virtual\s+)?'
    r'(?:void|int|float|double|string|auto|bool)\s+'
    r'(\w+)\s*\([^)]*\)\s*(?:const)?\s*\{',
    re.MULTILINE
)

# TODO/FIXME 统计
todo_re = re.compile(r'(TODO|FIXME|HACK|BUG)\b', re.IGNORECASE)

# 分支复杂度（简化圈复杂度）
branch_re = re.compile(r'\b(if|else|for|while|switch|case|catch)\b')

# 扫描所有 .cpp/.h 文件
for file in Path("./src").rglob("*.cpp"):
    content = file.read_text()
    funcs = func_re.findall(content)
    todos = todo_re.findall(content)
    complexity = len(branch_re.findall(content))
    print(f"{file.name}: {len(funcs)} funcs, {len(todos)} TODOs, complexity={complexity}")

度量维度一览

度量项正则模式用途
函数数量r'(\w+)\s*\([^)]*\)\s*\{'评估模块复杂度
类数量r'class\s+(\w+)'评估架构规模
TODO/FIXMEr'(TODO|FIXME|HACK)\b'追踪遗留问题
圈复杂度r'\b(if|else|for|while|switch)\b'评估可维护性
#includer'#include'评估依赖复杂度
代码行/注释行line.strip().startswith('//')评估注释率
dataclass：用 @dataclass 定义 FileMetrics 结构体，比手写 __init__ 简洁——这正是 Day 04 学过的 Python OOP 技巧。

4. 构建日志解析

C++ 编译日志动辄几百行，Python 正则一步提取所有 error/warning/note，然后 defaultdict 按文件分组统计。

Shell 管道方式
# grep 管道逐层过滤
grep "error:" build.log | wc -l
grep "warning:" build.log | sort -u

# 优点：快速
# 缺点：无法分类统计、无法生成报告
# 无法区分 gcc 和 MSVC 格式
Python 正则解析
# 一条正则匹配 gcc/clang 格式
error_re = re.compile(
    r'^(.+?):(\d+):(\d+): (error|warning|note): (.+)$',
    re.MULTILINE
)

# 自动分类+分组统计
from collections import defaultdict
by_file = defaultdict(lambda: Counter())

for m in error_re.finditer(log_text):
    file, line, col, level, msg = m.groups()
    by_file[file][level] += 1

# 还支持 MSVC 格式（备用正则）
# 一份脚本覆盖两大编译器

双编译器格式支持

编译器日志格式示例正则模式
gcc / clangmain.cpp:15:10: error: ...r'^(.+?):(\d+):(\d+): (error|warning): (.+)$'
MSVCmain.cpp(15): error C2065: ...r'^(.+?)\((\d+)\): (error|warning) (C\d+): (.+)$'
defaultdict(lambda: Counter())：嵌套 defaultdict + Counter，不用预先初始化空 Counter——比 C++ std::map<string, map<string, int>> 简洁得多，Day 02 学过的 Counter 真正发挥了威力。

5. 数据驱动的代码生成

这是 C++ 项目工具链的核心价值——从 JSON 规范自动生成 C++ 类，省去手写 getter/setter/构造函数的重复劳动。

PYTHON — JSON 规范定义数据模型
# 用 JSON 定义类的结构
spec = {
    "namespace": "app",
    "classes": [{
        "name": "User",
        "fields": [
            {"name": "id", "type": "int", "default": "0"},
            {"name": "name", "type": "std::string", "default": ""},
        ],
        "methods": ["toString", "validate"]
    }]
}
自动生成的 C++ 头文件
#pragma once
#include <string>
#include <sstream>

namespace app {

class User {
private:
    int id_ = 0;
    std::string name_ = "";

public:
    User() = default;
    explicit User(int id, std::string name) : id_(id), name_(name) {}

    int id() const { return id_; }
    void set_id(int val) { id_ = val; }
    const std::string& name() const { return name_; }
    void set_name(const std::string& val) { name_ = val; }

    std::string toString() const {
        std::ostringstream oss;
        oss << "id=" << id_ << ";";
        oss << "name=" << name_ << ";";
        return oss.str();
    }
};

} // namespace app

代码生成器核心逻辑

PYTHON — 生成头文件的核心循环
# 遍历每个字段 → 生成成员变量+getter+setter
for f in cls["fields"]:
    # 成员变量（C++ trailing underscore）
    h_lines.append(f"    {f['type']} {f['name']}_ = {f.get('default', '')};")

    # getter：std::string 用 const& 返回，int 直接返回
    if f["type"] == "std::string":
        h_lines.append(
            f"    const {f['type']}& {f['name']}() const {{ return {f['name']}_; }}")
    else:
        h_lines.append(
            f"    {f['type']} {f['name']}() const {{ return {f['name']}_; }}")
数据驱动思维：JSON 是数据模型（"你想生成什么"），Template 是渲染逻辑（"怎么生成"）。这种分离让代码生成器可维护、可扩展——加一个字段只需改 JSON，不改 Python 代码。

6. 配置驱动的增量构建

把构建参数从 Makefile/CMakeLists.txt 抽到 JSON，Python 读配置 → 动态构建命令 → subprocess 执行。

PYTHON — build_config.json
{
    "project_root": "./my_project",
    "targets": [
        {"name": "core", "sources": ["src/core.cpp"], "type": "library"},
        {"name": "app", "sources": ["src/main.cpp"], "type": "executable",
         "depends": ["core"]}
    ],
    "compiler": "g++",
    "flags": ["-std=c++17", "-Wall", "-O2"],
    "output_dir": "build"
}
CMake 构建方式
# CMakeLists.txt 定义构建规则
# cmake --build build/ 执行构建
# 优点：跨平台、成熟生态
# 缺点：规则固定，难以动态调整

add_library(core src/core.cpp)
add_executable(app src/main.cpp)
target_link_libraries(app core)
Python 配置驱动
# JSON 配置 → Python 动态构建命令
config = json.loads(Path("build_config.json").read_text())

for target in config["targets"]:
    cmd = f"{config['compiler']} {flags}"
    if target["type"] == "executable":
        deps = "-l".join(target["depends"])
        cmd += f" {sources} {deps} -o {name}"
    # subprocess.run(cmd, shell=True)
不是替代 CMake：Python 工具链补充 CMake，而非替代。常见分工：CMake 处理编译，Python 处理项目初始化、代码生成、日志分析、报告生成。两者搭配效果最佳。

7. 文件变更检测（哈希追踪）

增量构建的关键：判断哪些文件真正变了。CMake 用文件时间戳（touch 就会触发重建），Python 用 hashlib 内容哈希更精确。

时间戳方式（CMake）
# CMake 用文件修改时间判断
# 问题：touch file.cpp 就触发重建
# 即使文件内容没变

# C++  方式
auto mtime = std::filesystem::last_write_time(path);
// 仅比较时间戳，不比较内容
内容哈希方式（Python）
import hashlib

def file_hash(path):
    """SHA256 哈希 —— 内容变了才重建"""
    h = hashlib.sha256()
    h.update(Path(path).read_bytes())
    return h.hexdigest()[:16]  # 16位足够判断变更

# 对比新旧哈希
old = hash_cache.get(file_path)
new = file_hash(file_path)
if new != old:
    need_rebuild.append(file_path)  # 真正变了
哈希缓存：用 JSON 保存上次哈希（.build_hashes.json），下次构建时加载对比。这比 CMake 的 CMakeCache.txt 更轻量、更可读。

9. HTML 报告生成

所有度量、日志、测试数据最终汇总为一份深色主题 HTML 报告——用 Python 生成比手动整理快 10 倍。

PYTHON — HTML 报告生成核心
from string import Template

# 用 Template 定义报告模板
html_tpl = Template("""
<h1>📊 $name — 项目报告</h1>
<div class="stat-grid">
$stats_html
</div>
<h2>🔧 构建日志</h2>
$log_section
<h2>📁 文件统计</h2>
$file_table
""")

# 动态填充数据
stats_html = ""
for key, label in [("total_lines", "总行数"), ("functions", "函数数")]:
    stats_html += f'<div class="stat">{total[key]}</div>'

# 写入 HTML 文件
Path("report.html").write_text(html_tpl.substitute(
    name=project_name, stats_html=stats_html, ...
))
数据 → HTML：Python 把 JSON 数据渲染为 HTML，而 C++ 项目通常只有 Make 输出纯文本。深色主题 HTML 报告在浏览器中直观清晰，方便团队分享。

10. 完整工具链 CLI 集成

所有模块用 argparse 整合为一个 CLI 工具，类似 Day 09 的 cpp_project_builder.py，但覆盖更广。

PYTHON — argparse CLI 入口
import argparse

parser = argparse.ArgumentParser(
    description="C++ 项目综合工具链")
subparsers = parser.add_subparsers(dest="command")

# init 子命令
p_init = subparsers.add_parser("init", help="创建项目骨架")
p_init.add_argument("--name", required=True)
p_init.add_argument("--dir", default=".")

# gen 子命令
p_gen = subparsers.add_parser("gen", help="代码生成")
p_gen.add_argument("--spec", required=True)

# metrics 子命令
p_met = subparsers.add_parser("metrics", help="代码度量")
p_met.add_argument("--dir", required=True)

# report 子命令
p_rep = subparsers.add_parser("report", help="生成报告")
p_rep.add_argument("--dir", required=True)

使用示例

命令行用法
# 创建项目骨架
python cpp_toolchain.py init --name my_app --dir ./projects

# 代码生成
python cpp_toolchain.py gen --spec class_spec.json

# 代码度量
python cpp_toolchain.py metrics --dir ./src

# 构建日志解析
python cpp_toolchain.py log --file build.log

# 生成 HTML 报告
python cpp_toolchain.py report --dir ./my_app

# 演示模式（无需真实项目）
python cpp_toolchain.py --demo
argparse vs 手动 sys.argv：argparse 自动生成帮助信息、验证参数、支持子命令。C++ 的 CLI 解析通常用第三方库（如 CLI11），Python 标准库就有 argparse。
